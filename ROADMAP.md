# Tlias 前端项目企业级升级计划方案

## 一、项目现状评估

### 1.1 技术栈现状

| 维度 | 当前状态 | 评估 |
|------|----------|------|
| 框架 | Vue 3.2 + Vite 3.0 | 基础良好，Vue 3 版本可升级到 3.4+ |
| UI 组件库 | Element Plus 2.4（全量引入） | 可用，但全量引入体积大 |
| 状态管理 | 无（使用 localStorage） | 缺失，需引入 Pinia |
| 路由 | Vue Router 4.x + 静态路由 | 可用，但缺少动态路由和权限控制 |
| 请求库 | Axios + 基础拦截器 | 可用，需增强错误处理、取消请求等 |
| 类型系统 | 纯 JavaScript | 缺失，建议 TypeScript |
| 构建优化 | 基础配置 | 缺少拆包、压缩、性能分析 |
| 代码规范 | ESLint + Prettier | 基础可用，缺少 Git Hooks、提交规范 |
| 测试 | 无 | 完全缺失 |
| 监控 | 无 | 完全缺失 |

### 1.2 代码结构问题

1. **无状态管理**：用户信息、Token 直接操作 localStorage，组件间状态共享困难
2. **菜单硬编码**：菜单数据写在 layout 组件中，无法根据权限动态生成
3. **权限控制缺失**：只有登录态判断，无角色/按钮/数据级权限
4. **组件复用率低**：各页面的表格、搜索栏、表单弹窗重复代码多
5. **无统一工具函数**：日期格式化、金额格式化等散落在各页面
6. **无常量/枚举管理**：状态值、字典值硬编码在模板中

### 1.3 安全风险

1. **Token 存储在 localStorage**：易受 XSS 攻击
2. **无请求取消机制**：路由切换后请求仍在后台运行
3. **无防重复提交**：快速点击可能导致重复提交
4. **无操作审计**：关键操作无完整审计日志

---

## 二、升级目标与原则

### 2.1 总体目标

将项目从「能用的后台模板」升级为「可支撑百人团队协作、十万级用户的企业级系统」，在**安全性、可维护性、可扩展性、用户体验、运维效率**五个维度达到企业级标准。

### 2.2 核心原则

1. **渐进式升级**：不推翻重写，分阶段迭代，每个阶段都可独立上线
2. **最小破坏原则**：新增功能不影响现有业务，旧代码逐步迁移
3. **业务优先**：先解决阻碍业务发展的痛点，再做体验优化
4. **投入产出比**：优先做高收益、低成本的改造

## 三、分阶段实施计划

### 阶段一：基础夯实（P0 - 预计 2 周）

**目标**：解决最紧迫的架构问题，为后续扩展打基础

#### 任务 1.1：引入 Pinia 状态管理

**背景**：当前无状态管理，用户信息、Token 通过 localStorage 手动操作，组件间通信困难

**实施内容**：
- 安装 `pinia` + `pinia-plugin-persistedstate`（持久化插件）
- 建立 Store 目录结构：
  ```
  src/stores/
  ├── index.js          # Pinia 实例注册
  ├── modules/
  │   ├── user.js       # 用户信息、Token、权限
  │   ├── app.js        # 应用全局状态（侧边栏折叠、主题等）
  │   └── dict.js       # 数据字典缓存
  └── constants.js      # 常量定义
  ```
- 迁移现有 localStorage 操作到 user store
- 封装 useUser、useApp 等组合式函数

**验收标准**：
- [ ] 用户登录信息通过 Pinia 管理，自动持久化
- [ ] 退出登录、Token 过期等操作统一在 store 中处理
- [ ] 组件中不再直接操作 localStorage

#### 任务 1.2：完善 Axios 请求封装

**背景**：当前拦截器较基础，缺少取消请求、重试机制、请求并发控制等

**实施内容**：
- 新增请求取消机制（AbortController + 路由切换自动取消）
- 新增请求重试机制（网络错误自动重试 2 次，幂等请求）
- 新增重复请求拦截（相同参数的 pending 请求合并）
- 新增 Loading 管理（根据请求数量自动控制全局 loading）
- 完善错误码映射表，统一错误提示文案
- 新增接口白名单（不需要 Token 的接口）
- 新增请求/响应日志（开发环境控制台输出）

**验收标准**：
- [ ] 切换路由后，未完成的请求自动取消
- [ ] 网络波动时自动重试，不干扰用户
- [ ] 快速重复点击不会发出重复请求
- [ ] 错误提示文案统一，用户可理解

#### 任务 1.3：路由与权限体系搭建

**背景**：当前只有静态路由和简单登录判断，无法支持多角色权限

**实施内容**：
- 路由重构：
  - 静态路由（登录、404、403 等公共页面）
  - 动态路由（根据权限从后端获取或前端过滤）
- 实现 RBAC 权限模型：
  - 路由级权限（菜单是否可见）
  - 按钮级权限（自定义指令 `v-permission`）
  - 数据级权限（接口参数透传）
- 完善路由守卫：
  - Token 有效性校验
  - 权限不足跳转 403
  - 白名单机制
- 新增 403、404、500 等错误页面

**验收标准**：
- [ ] 不同角色登录看到不同菜单
- [ ] 无权限的路由直接访问会跳 403
- [ ] 按钮级权限通过 `v-permission="'emp:add'"` 控制
- [ ] 路由切换时有进度条提示

#### 任务 1.4：全局异常与错误边界

**背景**：当前缺少全局错误捕获，组件内错误可能导致白屏

**实施内容**：
- 新增全局错误处理（`app.config.errorHandler`）
- 新增 Vue 错误边界组件（ErrorBoundary）
- 错误上报接口对接（预留 Sentry 接入点）
- 统一的错误提示策略：
  - 接口错误：ElMessage 提示
  - 组件渲染错误：错误兜底 UI + 重试按钮
  - 严重错误：跳转到错误页

**验收标准**：
- [ ] 组件内 JS 错误不会导致整个应用白屏
- [ ] 错误有友好提示，用户知道发生了什么
- [ ] 开发环境有详细错误栈，生产环境有错误上报

### 阶段二：工程化与效率（P1 - 预计 3 周）

**目标**：提升团队协作效率，降低维护成本

#### 任务 2.1：TypeScript 迁移

**背景**：纯 JS 项目缺少类型检查，运行时错误多，重构风险大

**实施内容**：
- 配置 TypeScript 环境（tsconfig、类型声明文件）
- 类型文件目录结构：
  ```
  src/types/
  ├── api/            # 接口请求/响应类型
  ├── models/         # 业务模型类型
  ├── stores/         # Store 类型
  └── global.d.ts     # 全局类型声明
  ```
- 迁移策略：
  1. 先迁移工具函数、常量、类型定义
  2. 再迁移 API 层、Store
  3. 最后迁移组件（从公共组件到业务组件）
- 配置 `vue-tsc` 做类型检查

**验收标准**：
- [ ] 核心模块（API、Store、工具函数）100% 类型覆盖
- [ ] 新增代码必须使用 TypeScript
- [ ] `npm run type-check` 零错误通过

#### 任务 2.2：组件库封装与复用

**背景**：各页面重复代码多，如搜索栏、表格、表单弹窗模式高度一致

**实施内容**：
- 封装通用业务组件：
  - `ProTable`：高级表格（搜索栏 + 表格 + 分页 + 批量操作）
  - `ProForm`：高级表单（支持弹窗/抽屉/页面三种模式）
  - `DictTag`：字典标签（自动映射字典值到显示文本）
  - `ImageUpload`：图片上传组件
  - `FileUpload`：文件上传组件
- 组件目录结构：
  ```
  src/components/
  ├── common/         # 通用业务组件（与业务无关）
  └── business/       # 业务组件（与具体业务相关）
  ```
- 编写组件使用文档（README 或 Storybook）

**验收标准**：
- [ ] 新增列表页代码量减少 50% 以上
- [ ] 至少 3 个历史页面完成迁移验证
- [ ] 组件有完整的 Props/Events 文档

#### 任务 2.3：Git 工作流与代码规范

**背景**：当前缺少提交规范、代码检查卡点，代码质量不可控

**实施内容**：
- 配置 Husky + lint-staged：
  - `pre-commit`：运行 ESLint + Prettier 自动修复
  - `commit-msg`：校验提交信息格式
- 引入 Conventional Commits 规范：
  - feat: 新功能
  - fix: 修复 bug
  - docs: 文档更新
  - style: 代码格式
  - refactor: 重构
  - perf: 性能优化
  - test: 测试
  - chore: 构建/工具链
- 配置 `commitizen` 辅助生成规范提交信息
- 配置 CHANGELOG 自动生成（standard-version）

**验收标准**：
- [ ] 不符合规范的代码无法提交
- [ ] 不符合规范的 commit message 被拦截
- [ ] `npm run release` 可自动生成版本号和 CHANGELOG

#### 任务 2.4：环境配置与构建优化

**背景**：当前只有单一环境，缺少多环境配置和构建优化

**实施内容**：
- 多环境配置：
  - `.env.development` - 开发环境
  - `.env.test` - 测试环境
  - `.env.staging` - 预发布环境
  - `.env.production` - 生产环境
- 构建优化：
  - 路由级代码分割（已有，优化策略）
  - 第三方库拆包（vendor chunk）
  - Gzip 压缩（vite-plugin-compression）
  - 包体积分析（rollup-plugin-visualizer）
  - 图片压缩（vite-plugin-imagemin）
- 运行时配置：
  - `public/config.js` - 支持打包后修改接口地址等配置
  - 无需重新构建即可切换后端地址

**验收标准**：
- [ ] `npm run build:test` / `build:prod` 分别构建不同环境
- [ ] 生产构建首屏 JS 体积 ≤ 300KB（gzip 后）
- [ ] 打包后可通过修改 config.js 切换后端地址

## 四、关键技术决策

### 4.1 状态管理：Pinia vs Vuex

**决策：Pinia**

理由：
- Vue 3 官方推荐
- API 更简洁，TypeScript 支持更好
- 支持组合式 API 风格
- 不再有 mutations，代码更简洁

### 4.2 类型系统：渐进式 TypeScript

**决策：允许 JS/TS 共存，逐步迁移**

理由：
- 一次性全量迁移风险大、周期长
- 渐进式迁移可在每个迭代中完成一部分
- 新代码强制用 TS，旧代码按需迁移

### 4.3 UI 组件：Element Plus 按需引入

**决策：从全量引入改为按需引入 + 手动导入**

理由：
- 全量引入 gzip 后约 80KB，虽然不算特别大，但按需引入更优
- 使用 `unplugin-element-plus` 自动按需引入
- 配合手动导入常用组件，平衡开发效率和包体积

### 4.4 权限方案：前端路由过滤 + 接口鉴权

**决策：前端做菜单和按钮控制，后端做接口权限校验**

理由：
- 前端控制是体验优化，不是安全屏障
- 真正的权限控制必须在后端做
- 前端根据权限动态生成菜单和路由，提升用户体验

## 五、风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| TypeScript 迁移周期超预期 | 中 | 高 | 渐进式迁移，先核心后边缘，不追求 100% 覆盖 |

| 构建优化导致兼容性问题 | 中 | 低 | 主流浏览器覆盖测试，保留降级方案 |
