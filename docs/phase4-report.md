# 阶段四：监控与运维（P3）完成报告

## 一、阶段概述

本阶段聚焦于前端监控体系和运维能力建设，通过搭建性能监控、行为埋点、错误上报、日志管理等基础设施，为系统的稳定运行和持续优化提供数据支撑。同时完善了环境配置、构建信息注入等运维相关能力，提升项目的可维护性和可观测性。

## 二、完成任务清单

### 任务 4.1：前端监控体系 ✅

#### 4.1.1 性能监控

- **新增文件**：`src/monitor/performance.ts`
- **功能说明**：
  - **FP (First Paint)**：首次绘制时间
  - **FCP (First Contentful Paint)**：首次内容绘制时间
  - **LCP (Largest Contentful Paint)**：最大内容绘制时间
  - **CLS (Cumulative Layout Shift)**：累计布局偏移
  - **Navigation Timing**：完整的导航性能数据（DNS、TCP、TTFB、DOM Ready、Load）
  - **Resource Timing**：所有资源加载性能数据
- **技术实现**：
  - 使用 PerformanceObserver API 监听性能指标
  - 开发环境友好的控制台输出（带颜色和格式化）
  - 自动在页面加载完成后收集数据
  - 通过 CustomEvent 对外暴露数据

#### 4.1.2 行为埋点

- **新增文件**：`src/monitor/track.ts`
- **功能说明**：
  - **页面访问埋点**：自动记录页面停留时长
  - **点击埋点**：支持 `data-track` 属性声明式埋点
  - **自定义埋点**：手动调用 API 上报自定义事件
  - **批量上报**：达到阈值或定时批量上报
  - **离线安全**：使用 sendBeacon 确保页面关闭时数据不丢失
- **配置项**：
  - `enabled`：是否启用
  - `batchSize`：批量上报阈值
  - `throttleTime`：上报间隔
  - `maxQueueSize`：最大队列长度
  - `reportUrl`：上报接口地址

#### 4.1.3 监控统一入口

- **新增文件**：`src/monitor/index.ts`
- **功能说明**：
  - 统一初始化性能监控和行为埋点
  - 开发环境启动时打印监控状态
  - 对外暴露统一的 API
- **集成位置**：`src/main.ts` 中调用 `setupMonitor(app)`

## 三、架构设计

### 3.1 监控架构

```
┌─────────────────────────────────────────────────────┐
│                     监控层                            │
├──────────────┬──────────────┬───────────────────┤
│  性能监控      │  行为埋点      │  错误监控          │
│  (performance)│  (track)      │  (errorHandler)   │
├──────────────┴──────────────┴───────────────────┤
│                    日志系统 (logger)               │
├─────────────────────────────────────────────────────┤
│                    上报层                            │
│  (sendBeacon / CustomEvent / 后端上报接口)          │
└─────────────────────────────────────────────────────┘
```

### 3.2 日志分级策略

| 级别  | 开发环境输出 | 生产环境输出 | 是否上报 | 使用场景               |
| ----- | ------------ | ------------ | -------- | ---------------------- |
| DEBUG | ✅           | ❌           | ❌       | 调试信息，变量值输出   |
| INFO  | ✅           | ❌           | ❌       | 重要流程节点，状态变更 |
| WARN  | ✅           | ❌           | ❌       | 警告信息，非阻断问题   |
| ERROR | ✅           | ✅           | ✅       | 错误异常，功能不可用   |
| FATAL | ✅           | ✅           | ✅       | 致命错误，系统崩溃     |

### 3.3 数据上报策略

1. **批量上报**：日志队列达到 10 条或每 10 秒上报一次
2. **即时上报**：ERROR 及以上级别日志立即触发上报
3. **安全上报**：使用 `navigator.sendBeacon` 确保页面关闭时数据不丢失
4. **事件通知**：通过 CustomEvent 对外分发数据，便于接入自定义上报逻辑

## 五、技术亮点

### 1. 非侵入式设计

- 监控系统通过全局事件和拦截器工作，不侵入业务代码
- 行为埋点支持 `data-track` 声明式使用，业务代码零修改
- 日志系统可选启用，开发环境调试友好，生产环境无干扰

### 2. 性能友好

- 使用 PerformanceObserver 而非轮询，性能开销极小
- 日志队列限制最大长度，防止内存泄漏
- 批量上报 + 节流，减少网络请求频率
- sendBeacon 异步上报，不阻塞主线程

### 3. 开发体验优化

- 彩色日志输出，不同级别一目了然
- 应用启动时打印构建信息和监控状态
- 性能数据自动格式化（B/KB/MB、ms/s）
- 完整的 TypeScript 类型支持

### 4. 运维能力增强

- 多环境配置标准化
- 构建信息可追溯（版本、时间、环境）
- 代码分块策略清晰，便于缓存策略优化
- Gzip 压缩就绪，部署即可生效
